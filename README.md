# High-Order Markov Blanket Discovery

Code accompanying the paper:

> **High-Order Markov Blanket Discovery via a k-Order Relaxation of the Faithfulness Assumption**
> Loong Kuan Lee, Ragavi Krishnamoorthy, Nico Piatkowski.
> *Conference on Uncertainty in Artificial Intelligence (UAI), 2026.*

Constraint-based Markov-blanket (MB) discovery methods assume *faithfulness* and rely on
low-order conditional-independence (CI) tests, so they miss MB members that participate only in
**high-order interactions** (e.g. parity / XOR-type dependencies). **kOMB** (*k*-Order Markov
Blanket discovery) relaxes the faithfulness assumption to order *k*, recovering those members
while remaining tractable. The CI tests and the kOMB search are implemented in Rust for speed
(the `LIMMB` extension); the baselines come from [pyCausalFS](https://github.com/wt-hu/pyCausalFS).

## Repository layout

| Path | Contents |
| --- | --- |
| `LIMMB/` | Rust/PyO3 implementation of kOMB and the CI tests — builds the `LIMMB` Python module |
| `experiments/` | Data tooling and experiment drivers (see below) |
| `paper/` | LaTeX source of the paper; its tables and figures are regenerated by `analyse.py` |
| `pyCausalFS/` | Git submodule — baseline MB-discovery methods (BAMB, GSMB, HITON_MB, IAMB, LRH, MMMB, PCMB, STMB) |

## Installation

Prerequisites: **Python ≥ 3.10**, a **Rust toolchain** (<https://rustup.rs>), and a C compiler.

```bash
git clone https://gitlab.com/lklee/k-order-markov-blanket
cd k-order-markov-blanket

# Fetch the pyCausalFS baselines (git submodule)
git submodule update --init --recursive

# Create and activate a virtual environment
python -m venv .venv && source .venv/bin/activate

# Install Python dependencies and build the LIMMB Rust extension
pip install -r requirements.txt
```

The `-e ./LIMMB/` entry in `requirements.txt` builds the Rust extension automatically (via
[maturin](https://www.maturin.rs/)). If that step fails, build it directly:

```bash
pip install maturin
(cd LIMMB && maturin develop)
```

## Getting the data

Datasets are **not** committed. Fetch/generate them with:

```bash
python experiments/get_data.py           # the 4 paper datasets + synthetic data
python experiments/get_data.py --all     # all 11 real-world networks + synthetic data
```

* **Real-world** networks (Alarm1, Barley, Insurance, Mildew, …) are downloaded from the
  [MMHC supplement](https://pages.mtu.edu/~lebrown/supplements/mmhc_paper/mmhc_index.html)
  (Tsamardinos, Brown & Aliferis, *Machine Learning*, 2006) into `experiments/data/<Name>/`.
* **Synthetic** problems (parity, exactly-1, exactly-2, and, or) are generated locally by
  `data_gen.py`.

## Reproducing the results

Run the drivers from the `experiments/` directory (they use relative `./data`, `./res`,
`../pyCausalFS` paths):

```bash
cd experiments

# 1. Synthetic benchmarks (fast; F1 only, no runtimes) -> res/{func}_{size}/
python run_synthetic.py [--jobs N] [--skip-existing]

# 2. Real-world kOMB k–l ablation grid -> res/{dataset}/kOMB-{k}-{l}.pkl
#    Expensive (k=3 cells can be very slow); each target has a wall-clock cap.
python run_ablation.py [--jobs N] [--timeout SECS] [--datasets Alarm1 …] [--ks 1 2 3] [--ls 1 2 3]

# 3. Re-time the baselines on the same machine -> res/{dataset}/{METHOD}.pkl
#    Run AFTER step 2 (do not run concurrently — it would oversubscribe the cores).
python run_baselines.py [--jobs N] [--skip-existing]

# 4. Regenerate the paper's tables and figures from res/
python analyse.py
```

`analyse.py` writes LaTeX tables into `../paper/tab/*.tex` and figures into `../paper/fig/`, and
mirrors the underlying numbers to `./output/*.csv`. The real-world grid is the only heavy part
(many CPU-hours); the synthetic benchmarks run in seconds.
